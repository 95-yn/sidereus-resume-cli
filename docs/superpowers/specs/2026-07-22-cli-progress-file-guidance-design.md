# Resume CLI Loading 与文件路径提示设计

## 背景与目标

现有 CLI 能稳定解析简历、调用 AI 并输出结果，但耗时操作期间没有视觉反馈；当 PDF、JD 或显式环境配置路径错误时，也只报告文件不存在。用户难以判断程序是否仍在运行，输入路径出错后还需要自行查找文件。

本次改动提供两类体验优化：

1. 在交互式终端中显示分阶段 Loading 状态，同时保持 stdout 可被脚本安全消费。
2. 文件路径无效时，在受限范围内推荐最可能的文件，并给出可复制的重试提示。

不加入交互式文件选择、全盘搜索、OCR、自动采用候选文件或自动重试 API。

## Loading 交互

使用 `ora` 渲染动态 Spinner，并由项目内部的状态接口封装。业务命令不得直接操作 `ora`，以便测试和未来替换实现。

Spinner 只写入 stderr，并且仅在真实交互式终端中启用。CI、非 TTY、stdout 管道、测试环境或用户传入全局 `--no-progress` 时不输出 Loading 内容。业务结果仍只写入 stdout，因此 JSON、文本重定向和管道行为保持不变。

各命令使用以下阶段：

- `parse`：`正在读取并解析 PDF…`
- `extract`：`正在解析 PDF…` → `正在提取结构化信息…`
- `score`：`正在读取简历与 JD…` → `正在分析匹配度…`

成功时在 stderr 显示简短的 `完成` 状态。失败、API 超时、参数错误或进程中断时必须先停止并清理 Spinner，再由统一错误边界输出错误信息。mock 模式沿用相同阶段，但不增加人为延迟。

## 文件候选发现

新增独立的文件候选模块，输入包括用户路径、候选文件类型、当前工作目录和扫描限制，输出最多三个排序后的相对或绝对路径。该模块只读取目录项，不读取候选文件正文。

候选范围遵循以下规则：

- 优先扫描用户输入路径所在的已有父目录；父目录不存在时从当前工作目录开始。
- 最多向下扫描两层，并设置目录项总量上限，防止大目录拖慢错误处理。
- 排除 `.git`、`node_modules`、`dist`、覆盖率输出和其他常见生成目录。
- PDF 错误只推荐 `.pdf`；JD 优先推荐 `.txt`、`.md` 和 `.markdown`；显式 `--env-file` 推荐 `.env` 与 `.env.*`。
- 默认不进入隐藏目录；环境配置搜索允许匹配当前扫描目录中的 `.env` 文件。
- 根据文件名相似度、扩展名适配度、路径深度和大小写近似程度排序，稳定返回前三项。
- 如果用户传入的是目录，优先推荐该目录内符合类型的文件。

扫描失败或没有候选文件时，原始文件错误仍然可用；候选发现不得掩盖主要错误或改变退出码。

## 错误模型与终端文案

`AppError` 增加可选的结构化提示数据，包括候选路径、当前目录和参数角色。PDF、JD 与环境配置服务只负责产生错误事实，不拼接完整终端版式。顶层错误渲染器统一输出：

```text
错误：PDF 文件不存在：./cv.pdf

你可能想使用：
  1. ./简历/张三-cv.pdf
  2. ./resume.pdf

当前目录：/Users/example/interview
重试：resume-cli extract "./简历/张三-cv.pdf"
```

重试命令依据当前子命令和参数角色生成。含空格、中文或 shell 特殊字符的候选路径必须安全引用。没有候选时显示当前目录，并提示检查相对路径或使用绝对路径，不伪造重试命令。

建议仅用于展示：CLI 不自动选中、打开或读取其他候选文件，避免意外处理含隐私的简历或配置。

## 模块边界

- `src/utils/progress.ts`：定义状态接口和 `ora` 适配器，负责 TTY、CI、`--no-progress`、stderr 与清理行为。
- `src/utils/file-suggestions.ts`：执行受限目录扫描、类型过滤、相似度排序和最多三项截断。
- `src/utils/error-output.ts`：把 `AppError` 的结构化提示渲染为一致的终端文案，并生成安全引用的重试命令。
- `src/errors.ts`：扩展错误数据契约，但保持现有错误码和退出码。
- `src/services/pdf.ts`、`src/services/jd.ts`、`src/services/env.ts`：在路径类错误上附加候选发现所需的事实。
- `src/program.ts`：创建状态实例、识别当前命令上下文、传递 `--no-progress` 并确保退出前清理。
- `src/commands/`：只声明和切换业务阶段，不感知 TTY 或终端动画细节。

候选扫描和 Spinner 都通过明确接口注入或构造，使命令测试不依赖真实终端计时。

## 数据流

正常执行时，程序在命令 action 开始后启动第一阶段，命令在 PDF/JD 读取完成或进入 AI 调用前更新阶段，写出业务结果后标记完成。所有状态信息走 stderr。

路径错误时，服务确定主要错误码并请求候选模块查找相近文件，将结构化提示附加到 `AppError`。异常到达顶层后，程序先清理 Spinner，再由错误渲染器结合当前命令生成建议与重试示例，最后返回原有退出码 `2`。

非路径业务错误继续使用现有错误文案，不触发目录扫描。

## 测试策略

自动化测试覆盖：

- TTY 时 Spinner 启用；非 TTY、CI、测试默认值和 `--no-progress` 时关闭。
- Loading 仅写 stderr，stdout 的文本与 JSON 保持纯净。
- 三个命令的阶段顺序、成功状态以及任意阶段异常后的清理。
- PDF、JD 和 `.env` 候选的过滤、排序、深度、排除目录、扫描上限与最多三项。
- 中文、空格、大小写差异、目录输入、父目录不存在及无候选文件。
- 错误提示中的当前目录、候选编号、安全引用和重试命令。
- 原有错误码、退出码、mock、verbose、环境加载和输出文件行为不回归。

完成后运行 lint、类型检查、完整 Vitest、构建和 npm 打包检查，并用真实 tarball 安装到隔离 npm prefix，执行 `resume-cli --version`、`--help` 和至少一个错误路径冒烟测试。

## 分发与兼容性

新增 `ora` 作为运行依赖，并继续由 tsup 打入单文件 `dist/cli.js`。`package.json` 的 bin、prepare、prepack、files 白名单和 Node.js 20+ 要求不变。发布包不得包含 `.env`、测试、源码或候选扫描过程中发现的任何用户文件。
